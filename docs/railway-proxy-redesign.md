# Railway Proxy Redesign

## Goal

Make Railway backend a **stateless proxy** — all state lives in the daemon (trainer's Mac) with JSON files persistence. Backend becomes a thin bridge that rarely needs redeploying.

## Architecture Overview

```
Participant Browser                Railway (BE)                          Trainer's Mac
┌──────────────┐              ┌────────────────────────┐           ┌─────────────────┐
│              │── REST ─────>│  FastAPI Proxy          │           │                 │
│  HTML/JS/CSS │<─ static ────│  (stateless)            │<── WS ──>│  Daemon         │
│              │<─ WS ────────│                         │<── HTTP ──│  (FastAPI local) │
│              │              │  Static files (synced)  │           │  JSON on disk   │
│              │              │  Cached PDFs on disk    │           │                 │
│              │              │  Temp uploaded files    │           │                 │
└──────────────┘              └────────────────────────┘           └─────────────────┘

Host Browser ──── REST ───────────────────────────────────────────> Daemon localhost
```

## Key Design Decisions

1. **Only participant API calls are proxied** through Railway. Host web panel calls daemon localhost directly.
2. **Participant→BE WebSocket is receive-only** — all participant actions are REST calls (no WS messages from participant). WS is only for receiving daemon broadcasts.
3. **Daemon defines the participant OpenAPI** — daemon's FastAPI has a proper participant router with typed endpoints. The OpenAPI spec (`participant-api.json`) is uploaded to Railway on connect and served for inspection. Participant JS is written against this contract. Railway proxies blindly.
4. **File uploads** (static files) from daemon to BE use **HTTP POST**, not WS — more natural for binary data.
5. **WS between daemon↔BE** is for: proxying participant REST calls (request/response), and daemon→participant broadcasts (one-way push).
6. **AsyncAPI spec** will document the daemon↔BE WebSocket protocol.

## What lives where

### Railway BE (stateless, rarely deployed)
- Static files (HTML/CSS/JS) — synced from daemon via HTTP upload
- `participant-api.json` — OpenAPI spec uploaded by daemon
- PDF slides — downloaded from Google Drive directly by Railway (on daemon's command)
- Participant uploaded files — temporarily stored on disk (auto-deleted after 7 days)
- WS hub — fans out daemon broadcasts to all participants
- Generic REST proxy — forwards **participant-only** API calls to daemon over WS
- No auth, no business logic, no state

### Daemon (source of truth, on trainer's Mac)
- All app state (polls, votes, scores, Q&A, wordcloud, codereview, debate, leaderboard, sessions, participants)
- JSON files persistence on disk
- FastAPI server on localhost:
  - **Host panel** served directly (no proxy needed)
  - **All business logic** (scoring, timers, rate limiting, AI calls)
  - **Participant API** (called locally by BE proxy, or directly for testing)
- AI integrations (Claude API for code review smart paste, quiz generation)
- Summaries received from Claude Desktop (not AI-generated by the app)
- Google Drive slide downloads → pushes PDFs to BE via HTTP

## Communication Patterns

### 1. Participant REST calls (WS-proxied via BE)

```
Participant → BE:     POST /api/vote {option: "A"}
BE → Daemon WS:      {"type": "proxy_request", "id": "abc123", "method": "POST", "path": "/api/vote", "body": "..."}
Daemon calls localhost FastAPI, gets response
Daemon → BE WS:      {"type": "proxy_response", "id": "abc123", "status": 200, "body": "...", "content_type": "..."}
BE → Participant:     HTTP 200 {ok: true}
```

### 2. Host REST calls (direct to daemon)

```
Host → Daemon localhost:  POST /api/poll {question: "...", options: [...]}
Daemon processes, updates state, triggers broadcast
```

### 3. Daemon → Participant broadcasts (WS via BE)

```
Daemon → BE WS:           {"type": "broadcast", "message": {"type": "poll_opened", "poll": {...}}}
BE → All participant WS:  {"type": "poll_opened", "poll": {...}}
```

### 4. Participant WS (receive-only — event notifications from daemon)

```
Participant connects WS to BE — no messages sent by participant over WS.
BE → Participant WS:  {"type": "poll_opened", "poll": {...}}
BE → Participant WS:  {"type": "score_updated", "scores": {...}}
BE → Participant WS:  {"type": "qa_question_added", "question": {...}}
```

All WS messages are **semantic events** (something happened), not state dumps.

### 5. Participant state retrieval (REST, on load / reconnect)

```
Participant → BE → Daemon:  GET /api/session/state
Daemon responds with full current state (poll, activity, scores, Q&A, etc.)
```

Participant JS loads full state once on page load or WS reconnect, then applies
event deltas locally. No periodic state pushes. No opaque `type: "state"` messages.

### 5. File uploads (daemon → BE via HTTP)

```
Daemon → BE HTTP:  POST /internal/upload-static {"path": "participant.html", "content": "..."}
Daemon → BE HTTP:  POST /internal/upload-pdf   multipart file upload
Daemon → BE HTTP:  POST /internal/upload-openapi {"content": "...participant-api.json..."}
```

### 6. Static file sync (on daemon connect) + auto-reload

```
1. Daemon connects WS to BE
2. BE pushes:  {"type": "sync_files", "static_hashes": {"participant.html": "a3f2...", ...}, "pdf_slugs": {"deck-abc": "f1e2...", ...}}
3. Daemon compares static_hashes with local static/ folder
4. If static files differ → daemon uploads changed files via HTTP POST (pattern 5)
5. Daemon compares pdf_slugs with its slides catalog
6. If PDFs missing on BE → daemon sends download_pdf commands for each
7. If any static files were uploaded → daemon broadcasts "reload" event
8. Participant browsers receive "reload" event → refresh the page → load updated static files
```

This replaces the old `pending-deploy` mechanism. No deploy watcher needed — static file changes
are detected and propagated automatically on daemon (re)start.
PDF downloads are orchestrated by daemon but executed by BE directly from Google Drive.

## Broadcast Events (Daemon → BE → Participant WS)

Daemon sends `broadcast_event` over WS to BE. BE fans out the inner `event` to all participant WS connections. All events are semantic — they describe **what happened**, not raw state.

| Event type | Trigger | Payload |
|------------|---------|---------|
| `poll_opened` | Host opens poll for voting | `{poll}` |
| `poll_closed` | Host closes voting | `{vote_counts, total_votes}` — final results now visible |
| `poll_correct_revealed` | Host reveals answers | `{correct_ids}` |
| `poll_cleared` | Host deletes poll | `{}` |
| `poll_timer_started` | Host starts countdown | `{seconds, started_at}` |
| ~~`vote_update`~~ | ~~Participant votes~~ | **Removed** — vote counts included in `poll_closed` event; not shown during open voting |
| `activity_updated` | Host switches activity | `{current_activity}` |
| `qa_updated` | Any Q&A change (add, edit, delete, upvote, answer, clear) | `{questions}` — full questions list; small enough to send every time |
| `wordcloud_updated` | Participant or host submits a word, or host sets/changes topic | `{topic, words}` — full word list + current topic |
| `wordcloud_cleared` | Host clears wordcloud | `{}` |
| `codereview_opened` | Host creates review and opens line selection | `{snippet, language}` |
| `codereview_closed` | Host locks selection; review phase begins | `{}` |
| `codereview_line_confirmed` | Host confirms a line as correct | `{line}` |
| `codereview_cleared` | Host removes the code review | `{}` |
| `debate_launched` | Host creates debate | `{statement, phase}` |
| `debate_phase_updated` | Host advances phase | `{phase}` |
| `debate_side_update` | Participant picks side | `{sides_count}` |
| `debate_argument_added` | Participant submits argument | `{argument}` |
| `debate_argument_upvoted` | Participant upvotes | `{argument_id, upvote_count}` |
| `debate_ai_cleanup_done` | Daemon finishes AI cleanup | `{arguments}` |
| `debate_round_started` | Host starts timed round | `{round_index, seconds, first_side}` |
| `debate_round_ended` | Host ends round | `{}` |
| `debate_reset` | Host resets debate | `{}` |
| `leaderboard_revealed` | Host triggers leaderboard reveal; auto-fades after a few seconds | `{entries, total_participants}` |
| `scores_updated` | Scores changed (after poll correct, code review confirm, or reset) | `{scores}` — full scores map; each client displays its own by UUID. Reset = all zeros. |
| `slides_current_updated` | Current slide changed | `{slides_current}` |
| `slides_list_updated` | Slides catalog changed | `{slides}` |
| ~~`slides_cache_status`~~ | ~~PDF download progress~~ | **Removed** — PDF availability folded into `slides_list_updated`; each slide entry has an `available` flag |
| `summary_updated` | Summary changed | `{item_count, updated_at}` — lightweight notification; participant GETs full content on click |

| `session_ended` | Host ends session | `{}` |
| ~~`session_paused`~~ | ~~Host pauses session~~ | **Removed** — no pause concept |
| `deploy_pending` | New deploy detected | `{}` |

Participant JS applies these as deltas to local state. On page load or WS reconnect, client calls `GET /api/session/state` to get full current state.

## Daemon → BE WS Protocol (AsyncAPI)

### Messages: Daemon → BE

| Type | Purpose | BE action |
|------|---------|-----------|
| `proxy_response` | Response to a proxied participant REST call | Resolve pending future, return HTTP response |
| `broadcast_event` | Semantic event for all participants (e.g. `poll_opened`, `score_updated`) | Fan out identically to all participant WS connections |
| `set_active_session` | Daemon tells BE the current session ID (on connect + on change) | BE stores locally — used only for session validation on page load |
| `download_pdf` | Request BE to download a PDF from GDrive URL | BE downloads, saves to disk, responds with `pdf_downloaded` |

### Messages: BE → Daemon

| Type | Purpose | Daemon action |
|------|---------|---------------|
| `sync_files` | Sent on daemon connect — two inventories: (1) static file hashes `{path: hash}` and (2) cached PDF slugs on disk `{slug: hash}` | Daemon diffs static files, uploads changes via HTTP; diffs PDFs, requests missing downloads via `download_pdf` |
| `proxy_request` | Forwarded participant REST call | Call daemon localhost FastAPI, send `proxy_response` |
| `participant_connected` | New participant WS connected | Track in daemon state |
| `participant_disconnected` | Participant WS disconnected | Clean up in daemon state |
| `pdf_downloaded` | BE finished downloading a PDF | Daemon updates catalog, broadcasts slides list |

### Internal HTTP (Daemon → BE, not WS)

| Method | Path | Purpose |
|--------|------|---------|
| POST | `/internal/upload-static` | Upload/overwrite a static file |
| POST | `/internal/delete-static` | Delete a static file |

These `/internal/*` endpoints have no auth (Railway is not publicly exposing them — they're called by daemon over the internet, but could be protected by a shared secret header if needed).

## Resolved Design Decisions

### Broadcast model: ZERO personalization on BE
Analysis of all `broadcast_state()`, `broadcast_leaderboard()`, `broadcast_participant_update()`, and poll `result` messages confirms that **all per-participant data can be computed client-side**:
- `my_score` → client picks from scores map using own UUID
- `my_avatar`, `my_name` → client knows from initial set_name
- `voted_ids` → client tracked locally when it cast the vote
- `is_own` (Q&A) → client compares `author_uuid` with own UUID
- `has_upvoted` → client tracked locally when it sent upvote
- `your_rank` (leaderboard) → client computes from entries list + own UUID

**Result**: Daemon sends ONE broadcast message. BE fans it out identically to all participants. BE has zero personalization logic. Truly dumb pipe.

### Participant file upload → BE stores temporarily
- BE stores uploaded file as `{random_uuid}_{sanitized_filename}` in a flat folder (filename sanitized to prevent path traversal)
- Host downloads directly from Railway URL (UUID-obfuscated = security)
- BE auto-deletes files older than 5 days (background sweep)
- File entry persisted in session state permanently (associated with participant); download link stays visible even if file expired on disk (returns 404)
- BE notifies daemon via WS `file_uploaded` event on upload completion

### `/api/status` → BE-local only
- Returns `backend_version`, `daemon_connected` (WS exists?), `session_active`
- No proxy needed. Frontends get all state via WS broadcasts, not this endpoint.

### State persistence → JSON files (not JSON files)
- One JSON file per session folder (session state)
- One global JSON in sessions root (daemon global state)

### PDF slides: BE downloads, daemon orchestrates
PDFs must NOT transit through the trainer's Mac (bandwidth concern when traveling). The flow is:
1. On daemon WS connect: BE reports which PDF files exist on disk (`sync_static_files` includes PDF hashes)
2. Daemon knows the full slides catalog (from local PPTX→PDF conversion + GDrive URLs)
3. Daemon decides which PDFs the BE needs and sends: `{"type": "download_pdf", "slug": "...", "url": "https://drive.google.com/..."}`
4. BE downloads the PDF from GDrive directly, saves to disk
5. BE reports back: `{"type": "pdf_downloaded", "slug": "...", "success": true}`
6. Daemon broadcasts updated slides list to all participants
7. Participant requests `/api/slides/file/{slug}` → BE serves from disk (no proxy needed)

This keeps the orchestration logic in daemon while BE only does the dumb download + serve.

### Auth: reuse existing Basic Auth for everything
- Daemon WS handshake: already uses HOST_USERNAME/HOST_PASSWORD Basic Auth
- `/internal/*` HTTP endpoints: same Basic Auth header
- No new secrets needed

### Connection resilience: 5s grace period
- If daemon WS disconnects while participant REST calls are in-flight:
  - Wait up to 5s for daemon to reconnect
  - If reconnected within 5s: re-send pending requests (or let them complete if WS resumed)
  - If still disconnected after 5s: fail all pending requests with 503 "Trainer not connected"
- New participant requests while daemon is disconnected: queued for up to 5s, then 503

### Fields NOT persisted (transient, runtime-only)

These exist only in daemon memory, not in JSON files:

- `participants` WS connection map (runtime connections)
- `participant_history` set (rebuilt from `participants` keys in session state)
- `poll_timer_started_at` (timer is ephemeral; if daemon restarts, timer resets)
- `debate_round_timer_started_at` (same)
- `debate_ai_request` (pending AI work; lost on restart, host re-triggers)
- `quiz_request`, `quiz_refine_request`, `quiz_status`, `quiz_preview` (ephemeral quiz generation state)
- `slides_download_events`, `slides_gdrive_locks`, `slides_download_semaphore` (async infrastructure)
